

### 🤖 AGENT TASK: Create Unified `scrape_and_verify_links.py`

**Context for Agent:** We currently have two disparate scripts (`enrich_youtube_links.py` and `audit_youtube_links.py`) that perform two passes over our TMDB JSON files. This two-pass system is causing severe YouTube rate-limiting. Furthermore, some files have massive blocks of `null` values due to these rate limits (cascades at the beginning, middle, or end of files). 

**Your Task:**
Create a new script named `scripts/scrape_and_verify_links.py` that unifies the search and verification into a single `yt-dlp` call, and uses a smart threshold to only target broken data blocks.

**1. Smart Target Detection**
Write a function `get_films_to_process(films, threshold=12)` that returns a list of indices representing which films need scraping.
* **State 1 & 2 (Total Blank/Failure):** If the array has 0 valid `youtube_link`s (either the key is missing entirely, or every single link is `null`), return *all* indices.
* **State 3 (Cascades & Blocks):** Iterate through the array and find contiguous blocks of missing or `null` links. If a contiguous block's length is `>= threshold`, add the indices of *that specific block* to the return list. Ignore smaller natural gaps (e.g., blocks of 1 to 11 nulls).

**2. The Unified Scrape & Verify Logic**
For the identified films, do not use `requests` and regex. Use `yt-dlp` directly with the `ytsearch1:` parameter to grab the first search result and its metadata in one move.
* **Search Query:** `f"ytsearch1:{film_title} {composer_name} main theme suite"`
* **Verification (Check A & B):** Immediately run the validation logic from the old audit script (Normalize strings -> Check A: Film title in YT title? -> Check B: Film title AND Composer last name in YT description?).
* **Action:** If it passes, set `film['youtube_link'] = result_url`. If it fails, set `film['youtube_link'] = None`. 

**3. Robust Rate-Limit Handling & Checkpointing**
* Wrap the `yt-dlp` call in a retry loop (max 4 retries).
* **CRITICAL:** Catch rate limit exceptions (HTTP 429, "try again later", "rate-limited"). If hit, trigger exponential backoff (e.g., start at 5 minutes, double each retry) before trying the *same* film again.
* Write to the JSON file immediately after a successful update or a definitive `None` assignment to ensure we don't lose progress if the script is force-killed.

**4. Execution**
Once written, execute the script over the `data/filmographies/` directory.
